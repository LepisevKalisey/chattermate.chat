# Premortem-анализ развертывания ChatterMate на Coolify

Этот документ содержит анализ рисков деплоя и потенциальных причин сбоев (premortem), которые могут помешать стабильной работе ChatterMate на домене `chat.lexi.kz` в окружении Coolify.

---

## 🔍 Сценарии потенциальных сбоев и меры по их предотвращению

### 1. Ошибка при подключении к базе данных: `NoSuchModuleError: Can't load plugin: sqlalchemy.dialects:postgresql`
* **Сценарий сбоя:**
  В Coolify база данных автоматически генерирует строку подключения `DATABASE_URL` в формате `postgresql://user:pass@host:5432/db`. 
  Поскольку в бэкенде установлен драйвер `psycopg` (версии 3), а не устаревший `psycopg2`, SQLAlchemy 2.0 при попытке обработать протокол `postgresql://` выбросит исключение и упадет, так как по умолчанию ожидает наличие `psycopg2`.
* **Вероятность:** 🔴 **Критическая (100% при дефолтном URL от Coolify)**
* **Предотвращение:**
  Добавлен Pydantic-валидатор в [config.py](file:///c:/Projects/chattermate.chat/backend/app/core/config.py), который автоматически перезаписывает протоколы `postgresql://` и `postgres://` на `postgresql+psycopg://`. Это делает подключение совместимым с установленной библиотекой `psycopg` v3.

---

### 2. Сбой при инициализации базы данных: `role "postgres" does not exist` в `init-vector.sql`
* **Сценарий сбоя:**
  В файле [init-vector.sql](file:///c:/Projects/chattermate.chat/backend/scripts/init-vector.sql) жестко прописаны команды:
  `GRANT USAGE ON SCHEMA ai TO postgres;`
  Если пользователь меняет имя суперпользователя базы данных в Coolify с помощью `POSTGRES_USER` на что-то более безопасное (например, `chattermate_prod`), роль `postgres` не создается. В результате скрипт инициализации падает с ошибкой, блокируя успешную настройку расширения `pgvector`.
* **Вероятность:** 🟠 **Высокая**
* **Предотвращение:**
  Поскольку скрипт выполняется от имени владельца БД/суперпользователя, ему не требуются явные гранты на им же созданную схему `ai`. Гранты роли `postgres` будут обернуты в проверку существования роли либо заменены на `CURRENT_USER`, чтобы исключить падение скрипта.

---

### 3. Бесконечное скачивание модели FastEmbed при каждом рестарте контейнера
* **Сценарий сбоя:**
  Модель эмбеддингов `BAAI/bge-small-en-v1.5` используется библиотекой `fastembed` через Agno. По умолчанию `fastembed` скачивает модель в домашнюю директорию пользователя (`~/.cache/fastembed_cache`). 
  В отличие от Hugging Face и PyTorch, для `fastembed` в [docker-compose.coolify.yml](file:///c:/Projects/chattermate.chat/docker-compose.coolify.yml) не была настроена переменная кэша и примонтированный volume. При каждом обновлении или перезапуске контейнера модель будет скачиваться заново, замедляя старт на несколько минут и создавая избыточную нагрузку на сеть.
* **Вероятность:** 🟠 **Высокая**
* **Предотвращение:**
  В `docker-compose.coolify.yml` для сервисов `backend` и `knowledge_processor` будет добавлена переменная окружения `FASTEMBED_CACHE_DIR=/app/.cache/huggingface/fastembed`. Это перенаправит кэш моделей FastEmbed внутрь уже примонтированного именованного тома `huggingface_cache`.

---

### 4. Потеря авторизации Firebase при обновлении кода (эфемерный bind mount)
* **Сценарий сбоя:**
  Бэкенд монтирует локальную директорию `./config` в `/app/config` для хранения файла конфигурации `firebase-config.json`.
  В Coolify директория сборки (`/artifacts/<uuid>/`) очищается при каждом новом коммите и деплое. Если конфигурационный файл был загружен напрямую в контейнер или создан вручную на сервере в папке `./config`, он будет удален при следующем пуше в репозиторий.
* **Вероятность:** 🟡 **Средняя** (Firebase не используется по умолчанию, но критично при активации)
* **Предотвращение:**
  Рекомендуется передавать конфигурационные файлы через переменные окружения (например, JSON-строку в `FIREBASE_CREDENTIALS` с парсингом внутри приложения) или использовать named volume для персистентного хранения папки `/app/config`.

---

### 5. Сбой синхронизации CORS при обрыве соединения с Redis
* **Сценарий сбоя:**
  В [cors.py](file:///c:/Projects/chattermate.chat/backend/app/core/cors.py) фоновый слушатель обновлений CORS `listen_for_cors_updates` при возникновении сетевой ошибки с Redis переходит в вечный цикл сна на 1 секунду, опрашивая тот же самый сломанный объект подписки `pubsub`. Он не переподключается к Redis автоматически.
* **Вероятность:** 🟡 **Средняя**
* **Предотвращение:**
  Поскольку ChatterMate в Coolify развертывается с одним воркером (`WORKERS=1`), CORS-настройки обновляются синхронно внутри этого процесса. Наличие сетевых проблем с Redis не нарушит CORS для текущего контейнера. Однако при масштабировании на несколько воркеров потребуется доработка логики переподключения слушателя.

---

### 6. Превышение таймаута сборки (Build Timeout) на слабых серверах
* **Сценарий сбоя:**
  Docker-образ бэкенда весит около 3–5 ГБ (из-за PyTorch, Playwright и компиляции зависимостей). 
  Если на сервере Coolify параллельно запущено много других служб, одновременная сборка или повторная сборка контейнеров может занять более 10–15 минут, что приведет к таймауту сборки в Coolify (Exit Code 255/137).
* **Предотвращение:**
  Мы уже оптимизировали `docker-compose.coolify.yml`, убрав дублирующуюся сборку из `knowledge_processor` (теперь он использует готовый образ бэкенда `chattermate-backend`). Это снижает нагрузку на CPU/RAM вдвое в процессе сборки.
